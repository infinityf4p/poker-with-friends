# Poker with Friends v0.2.0

Released: 2026-07-12

## Highlights

- Persistent user accounts replace disposable display-name-only identity.
- Every eligible player must confirm each hand; no countdown can silently start the next hand.
- Room administrators can manage membership and perform state-constrained chip issue/collection operations with an auditable ledger.
- Member removal and restoration are explicit administrative actions rather than implicit balance edits.
- The table always identifies the acting player and every seat's poker position, including heads-up `BTN/SB`.
- Mobile and desktop action surfaces include exact `amountTo`, Open/3-Bet/4-Bet, post-flop pot fractions, Call, Check, Fold and All-in controls.
- Cards render ten as `10`, hand history is organized by player action and street, and motion respects `prefers-reduced-motion`.
- Betting fixes cover large opening-bet minimum raises, short all-in reopening rules and short-big-blind bring-in behavior.
- Production deployment now uses a generic Compose stack with an independent PostgreSQL service, named database storage, and configurable application bind address and port.
- Public-repository CI covers formatting, type checking, tests, builds, container builds, secret scanning, and CodeQL.

## Deployment changes

- Use `infra/compose.yaml` as the canonical production definition.
- Copy `infra/.env.production.example` to `infra/.env`; do not reuse development secrets.
- `SNAPSHOT_KEY` must be Base64 encoding of exactly 32 random bytes.
- `APP_BIND_ADDRESS` and `APP_BIND_PORT` replace any host-specific published-port configuration.
- PostgreSQL data uses the Compose-managed `postgres-data` volume by default; `POSTGRES_DATA_PATH` supports an operator-managed data disk.
- Use an immutable release tag or Git SHA for `APP_IMAGE_TAG` and record the same SHA in `APP_BUILD_SHA`.

## Upgrade notes

The identity model changed from disposable room sessions to permanent accounts. Upgrading an existing v0.1 database in place is not supported by this release. Preserve a final backup for audit, reset the application schema, run all migrations against the empty database, and create player accounts again from the administrator dashboard.

1. Stop writes and create a verified final v0.1 backup.
2. Reset only this application's PostgreSQL database. If reusing the database, drop and recreate the application-owned `public` schema and also drop the application-owned `drizzle` schema so stale migration records cannot suppress the v0.2 migration chain.
3. Build the v0.2.0 image and run the complete migration chain.
4. Sign in with the configured administrator and recreate player accounts and rooms.
5. Verify login, membership, chip ledger, WebSocket reconnect, manual hand confirmation, and both game modes before reopening access.

Do not start an older application image against a schema marked non-reversible. Restore the corresponding database backup when schema rollback is required.

## Product boundary

Poker with Friends continues to use non-redeemable entertainment points. This release adds no payment, cash-out, rake, escrow, or real-money settlement feature. Operators remain responsible for privacy, age restrictions, and local legal compliance.
